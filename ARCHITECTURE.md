# 架构

## 高层级结构

自在音乐是一款单进程 Linux 终端界面（TUI）应用。核心职责被拆分到多个专注的 Zig 模块中：

- `src/main.zig`：初始化内存分配器，并将控制权移交至应用层。
- `src/app.zig`：管理交互模型、事件循环、渲染、按键处理，以及扫描、存储和播放模块间的协调工作。
- `src/library.zig`：扫描配置的音乐根目录、提取元数据，并生成播放列表视图。
- `src/playback.zig`：将 `libmpv` 封装为面向队列的控制器，供界面层安全调用。
- `src/storage.zig`：将扫描后的音乐库数据和播放会话状态存储至 SQLite 数据库。
- `src/config.zig`：解析 XDG 规范路径并加载配置文件。
- `src/domain.zig`：定义全项目共享的数据模型。

## 运行时数据流

### 启动流程

1. `main()` 函数创建内存分配器，并调用 `app.run()`。
2. `app.Model.init()` 从 `config.zig` 加载配置和 XDG 路径。
3. 应用通过 `storage.zig` 打开 SQLite 缓存。
4. 缓存的音乐库数据被加载至基于内存池（arena）的 `domain.Library` 结构体中。
5. 播放控制器通过 `playback.zig` 完成初始化。
6. 终端界面（TUI）事件循环启动，并可能触发一次异步的初始扫描。

### 扫描与重载

`library.zig` 负责全量扫描和增量重载：

- 文件系统发现：遍历并收集候选音频文件路径。
- FFmpeg 元数据探测：提取标题、艺术家、专辑、时长等信息。
- 元数据降级策略：当音频标签不完整时，从文件路径生成兜底元数据。
- 播放列表重建：基于文件夹、艺术家、专辑维度重新构建播放列表。

应用层在后台线程执行扫描，并通过共享的扫描状态和应用事件交换进度。这确保扫描过程中界面仍能保持响应，同时扫描进度和完成提示会实时渲染在界面上。

### 播放逻辑

`playback.zig` 持有 `libmpv` 句柄和音频文件路径队列。应用层不直接调用 `libmpv`，而是通过控制器发起以下操作：

- 从指定音轨加载播放队列。
- 播放队列中指定位置的音轨。
- 暂停/恢复播放。
- 切换至下一曲/上一曲。
- 切换播放模式（单曲循环/列表循环/随机等）。

这种设计将 mpv 相关的具体逻辑与界面状态机隔离开，降低耦合。

### 持久化存储

`storage.zig` 管理两类数据的持久化：

- 音乐库缓存：
  - 音轨信息
  - 派生的播放列表
  - 播放列表与音轨的关联关系
- 轻量级会话状态：
  - 当前播放模式
  - 当前播放音轨路径
  - 当前播放进度

界面的临时导航状态则刻意不做持久化处理。

## 构建期界面字符串生成

界面可见的文本内容统一维护在 `ui_strings.json` 文件中。

在执行 `zig build` 构建时，`build.zig` 会执行以下步骤：

1. 读取 `ui_strings.json` 文件。
2. 解析 JSON 内容。
3. 生成 `ui_strings.zig` 源文件并加入构建依赖图。
4. 将其暴露为 `ui_strings` 模块供项目引用。

运行时代码通过 `@import("ui_strings").Strings` 导入字符串常量，因此修改界面文本仅需编辑 JSON 文件并重新构建即可。

## 外部依赖

- `libvaxis`：终端界面渲染、输入处理与事件管理。
- `libmpv`：音频播放后端。
- `sqlite3`：缓存数据与会话状态的持久化存储。
- FFmpeg（`libavformat`、`libavcodec`、`libavutil`）：音频元数据提取。

这些依赖均在 `build.zig` 中以系统库形式链接到项目。

---

### 总结

1. 自在音乐采用模块化设计，将界面、播放、扫描、存储等核心能力拆分到独立 Zig 模块，降低耦合且职责清晰。
2. 运行时通过后台线程执行扫描保证界面响应，播放逻辑通过封装 `libmpv` 隔离底层细节，提升可维护性。
3. 界面文本通过 JSON 管理+构建期生成 Zig 代码的方式，实现文案与代码解耦，便于多语言/文案调整。
